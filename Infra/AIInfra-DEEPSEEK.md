# AI 基础设施深度问答

## 一、GPU 与 CPU 在 AI 训练中的本质区别

### 1. 架构设计哲学完全不同

**CPU（中央处理器）** 是为 **串行任务** 优化的：
- 几个（4-32 个）强大、复杂的核心
- 每个核心都有独立的分支预测、乱序执行、大容量缓存
- 擅长处理复杂的控制逻辑、条件分支、多任务切换
- 类比：少数几个博士，每个都能独立解决复杂问题

**GPU（图形处理器）** 是为 **并行任务** 优化的：
- 数千个简单、小型的核心（如 NVIDIA H100 有 16896 个 CUDA 核心）
- 所有核心执行相同的指令，处理不同的数据
- 擅长做大量简单、重复的数学运算
- 类比：成千上万个小学生，每人只做简单加减法，但一起做极快

### 2. 深度学习计算的本质

深度学习的核心运算就是 **矩阵乘法** 和 **向量运算**：

```
一个 Transformer 的前向传播 ≈ 大量的矩阵乘法
输入: [batch_size, seq_len, hidden_dim]
权重: [hidden_dim, hidden_dim]
输出 = 输入 × 权重  (这就是矩阵乘法)
```

矩阵乘法的特点：
- **数据并行**：每个输出元素的计算是独立的
- **计算密集**：一次矩阵乘法涉及海量的乘加运算
- **控制简单**：没有复杂的分支、跳转逻辑

这恰好完美匹配 GPU 的 SIMD（单指令多数据）架构。

### 3. 关键数据对比

以 NVIDIA H100 为例：

| 指标 | 高端 CPU (如 AMD EPYC) | NVIDIA H100 GPU |
|------|------------------------|-----------------|
| 核心数 | ~96 核 | 16896 CUDA 核心 |
| FP16 算力 | ~2 TFLOPS | ~1979 TFLOPS (稠密) |
| 内存带宽 | ~460 GB/s | ~3.35 TB/s |
| 功耗 | ~300W | ~700W |

**GPU 的 FP16 算力是 CPU 的约 1000 倍。**

### 4. 为什么 CPU 替代不了 GPU

即使你有 1000 个 CPU 核心，也很难替代一个 GPU，原因：

- **内存带宽瓶颈**：训练时每步都要把权重、梯度、优化器状态搬来搬去，CPU 的内存带宽差 GPU 一个数量级
- **通信开销**：多 CPU 之间的通信延迟远高于 GPU 内部共享内存
- **指令开销**：CPU 每个核心要独立取指、解码，GPU 是单指令多线程（SIMT），一个指令管几千个线程

### 5. 未来趋势

- **专用芯片崛起**：TPU（Google）、NPU 等 ASIC 芯片比 GPU 更"极端"，直接为矩阵乘法定制电路
- **GPU 也在进化**：Tensor Core 专门加速矩阵乘加（MMA），比传统 CUDA 核心又快 8-16 倍
- **混合计算**：推理场景下，Apple M 系列的统一内存架构结合 NPU，也在缩小差距，但训练仍然是 GPU 的天下

**一句话总结**：深度学习 90% 的计算是矩阵乘法，矩阵乘法天生可并行，GPU 天生为并行而生——这是完美的供需匹配。

## 二、FLOPS、TFLOPS、内存带宽与屋顶线模型

### FLOPS 是什么

**FLOPS** = **F**loating-point **O**perations **P**er **S**econd（每秒浮点运算次数），衡量硬件的**计算能力上限**。

```
一次浮点运算 = 加、减、乘、除任意一种
FLOPS = 硬件1秒钟能做多少次这样的运算
```

常见前缀：

| 单位 | 含义 | 举例 |
|------|------|------|
| GFLOPS | 10⁹ FLOPS | iPhone 的 NPU |
| TFLOPS | 10¹² FLOPS | NVIDIA H100：1979 TFLOPS (FP16) |
| PFLOPS | 10¹⁵ FLOPS | 大型超算集群 |

**注意 FP16 vs FP32**：H100 标 1979 TFLOPS 是指 **FP16 稀疏算力**，实际 FP32 只有 67 TFLOPS。厂商宣传数字要看精度。

### 内存带宽

**内存带宽** = 数据在计算单元和内存之间搬运的**最高速率**（单位 GB/s 或 TB/s）。

```
内存带宽 = 内存频率 × 总线位宽 × 通道数
```

以 H100 为例：
```
HBM3 显存：3.35 TB/s 带宽
普通 DDR5：~50 GB/s
差距：约 67 倍
```

### 屋顶线模型

屋顶线模型是一个二维图，用来判断一个算法是 **算力瓶颈** 还是 **带宽瓶颈**。

```
算力
↑
│  ┌──────────────────────────── 算力屋顶（硬件上限）
│  │        算力瓶颈区
│  │    (矩阵乘法、大卷积)         │
│  │         斜率 = 内存带宽      /
│  │  带宽瓶颈区                  /
│  │ (逐元素操作、ReLU、Dropout) /
│  └───────────────────────────
│                              ↑
│                         这里斜率是内存带宽
└──────────────────────────────────→ 计算强度 (FLOPS/Byte)
```

- **X 轴：计算强度** = 每读 1 byte 数据能做多少次浮点运算
- **Y 轴：可达算力** = 硬件实际能达到的算力上限
- **斜线（带宽瓶颈线）**：Attainable FLOPS = 计算强度 × 内存带宽
- **水平线（算力瓶颈线）**：Attainable FLOPS = 硬件峰值 FLOPS
- **屋顶线 = 这两条线的最小值**

以 H100 为例（FP16 稠密算力 990 TFLOPS，带宽 3.35 TB/s）：
```
转折点 = 990 TFLOPS / 3.35 TB/s ≈ 295 FLOPS/Byte
计算强度 < 295 → 带宽瓶颈
计算强度 > 295 → 算力瓶颈
```

常见操作的分类：

| 操作 | 计算强度 | 瓶颈 |
|------|---------|------|
| ReLU / Dropout | 极低 | 带宽 |
| LayerNorm | 低~中 | 带宽 |
| 小卷积（1×1, 3×3） | 较低 | 通常带宽 |
| 大矩阵乘法（GEMM） | 高 | 算力 |
| Flash Attention | 中（优化后） | 介于两者 |

## 三、分布式训练 3D 并行 (DP/TP/PP)

### 训练大模型时的三个瓶颈

1. **显存放不下**：模型参数 + 梯度 + 优化器状态 > 单卡显存
2. **算力不够**：单卡算得动但太慢
3. **通信开销**：卡越多，通信占比越高，加速比下降

### DP（数据并行）—— 最简单的加速

每张卡放一份完整模型，数据分成 N 份分发到 N 张卡，各自前向+反向算出梯度，然后对梯度做 AllReduce 取平均。

```
卡0: 模型副本 + 数据 batch[0:32]  → 梯度0
卡1: 模型副本 + 数据 batch[32:64] → 梯度1  → AllReduce(梯度) → 更新
卡2: 模型副本 + 数据 batch[64:96] → 梯度2
```

- 解决的问题：数据量大、单卡算不过来
- 解决不了的：单卡显存放不下整个模型
- 通信特点：需要 AllReduce 梯度，通信量与参数总量成正比

### TP（张量并行）—— 拆单层算力

把一层内的权重矩阵 **竖着切** 或 **横着切**，分到多张卡上。每张卡算矩阵乘法的一部分。

```
普通矩阵乘法：Y = X × W

TP 列切法：
  把 W 切成 [W1 | W2]
  卡0 算 Y1 = X × W1
  卡1 算 Y2 = X × W2
  最后拼起来 Y = [Y1 | Y2]
```

- 解决的问题：模型单层太大，单卡显存放不下或算不完
- 代价：前向/反向都需要通信同步，通信量大且频繁
- 适用条件：卡间必须有 **高速互联**（NVLink, NVSwitch）

### PP（流水线并行）—— 按层切分

把模型按层切成 N 段，每张卡放一段。数据依次流过。

```
卡0: Embedding + Transformer 0-7
卡1: Transformer 8-15
卡2: Transformer 16-23
卡3: Transformer 24-31 + LM Head

前向：卡0 → 卡1 → 卡2 → 卡3
反向：卡3 → 卡2 → 卡1 → 卡0
```

- 解决的问题：模型层数多，单卡显存放不下
- 代价：**流水线气泡**——启动和结束时有卡空闲
- 优化手段：GPipe（分批注入微 batch）、1F1B（交错 forward/backward）

### 三者对比

| 维度 | DP | TP | PP |
|------|----|----|-----|
| 切什么 | 数据 | 单层内的权重 | 按层切分 |
| 单卡显存 | 存完整模型 | 存切后权重 | 存部分层 |
| 通信频率 | 每 step 一次 | 每层多次 | 层边界一次 |
| 通信量 | 中等（梯度） | 最大 | 最小（激活值） |
| 卡间要求 | 普通网络即可 | 必须 NVLink | 普通网络可接受 |

### 3D 并行组合策略（从内到外）

1. **最内层 TP**：8 张卡通过 NVSwitch 做张量并行，通信量最大必须放在高速域内
2. **中层 PP**：同节点内或相邻节点做流水线并行，通信量小
3. **最外层 DP**：大量节点做数据并行，Gradient AllReduce

典型配置（LLaMA 405B 训练）：
```
总共 4096 张 H100
TP = 8  (同一个 HGX 底座内的 8 卡 NVSwitch)
PP = 4  (模型切成 4 段)
DP = 4096 / (8×4) = 128 个数据并行副本
```

选型指南：
```
中等模型 (7B-13B)：DP 就够了，或者 DP + FSDP
大模型 (70B)：     DP + FSDP，或 TP + DP
超大模型 (405B+)： TP + PP + DP 三合一
```

## 四、ZeRO-1/2/3/Offload 原理与内存节省

### 训练显存构成

用 Adam 优化器训练时（**K = 12**）：

```
总显存 = Model States + Residual States

Model States:
  ├── 参数 (Φ, FP16)
  ├── 梯度 (Φ, FP16)
  └── 优化器状态 (12Φ): 动量m(4Φ) + 方差v(4Φ) + master params(4Φ)

Φ=7B 参数: Model States = 14 × 7B = 98 GB
```

### ZeRO-1：拆分优化器状态

优化器状态 (KΦ) 切成 N 份，每张卡只存 1/N。

```
ZeRO-1 内存 = (2 + K/N) × Φ = 2 + 12/N → N=64 时 ≈ 2.19Φ
通信量：与 DP 相同，无额外开销
```

### ZeRO-2：拆分梯度 + 优化器状态

梯度和优化器状态各切成 N 份。

```
ZeRO-2 内存 = (2 + (1+K)/N) × Φ = 2 + 13/N → N=64 时 ≈ 2.20Φ
通信量：与 DP 相同
```

### ZeRO-3：连参数也拆分

参数、梯度、优化器状态全部切成 N 份，按需从其他卡 AllGather。

```
ZeRO-3 内存 = (2+K)/N × Φ = 14/N → N=64 时 ≈ 0.22Φ
通信量：比 ZeRO-1/2 大一倍（每层前向 AllGather + 反向 AllGather）
```

Φ=7B 用 64 卡 ZeRO-3：Model States 从 98GB → 约 3GB，省了约 97%。

### ZeRO-Offload：借 CPU/NVMe

将优化器状态和梯度卸载到 CPU 内存，GPU 只保留参数和激活值。

```
GPU 显存 ≈ Φ/N + 激活值
CPU 内存 ≈ (KΦ + Φ)/N
```

### 对比总览

| 方案 | 切分什么 | Model States 内存 | N=64 时 | 额外通信 |
|------|---------|-------------------|---------|---------|
| 原生 DP | 无 | (2+K)Φ = 14Φ | 14Φ | 基准 |
| ZeRO-1 | Optimizer | 2+K/N | 2.19Φ | 无 |
| ZeRO-2 | Opt + Grad | 2+(1+K)/N | 2.20Φ | 无 |
| ZeRO-3 | 全部 | (2+K)/N | **0.22Φ** | +100% |
| ZeRO-Offload | + 搬到 CPU | GPU: Φ+Activations | GPU ≈ Φ | PCIe 搬运 |

### 选型

```
显存紧张但不严重 → ZeRO-1/2（通信少）
显存刚好但需要大 batch → ZeRO-2
显存完全放不下 → ZeRO-3
ZeRO-3 还不够 → ZeRO-3 + Offload
```

## 五、PagedAttention 原理

### 传统 KV Cache 的痛点：显存碎片化

KV Cache 长度动态变化但显存静态预分配 max_len，导致内部+外部碎片，利用率仅 20%-40%。

```
┌──────────┬──空洞──┬────────────┬──空洞──┬──────┐
│ 请求1(300)│ (浪费)  │ 请求2(1500) │ (浪费)  │ 请求3 │
└──────────┴────────┴────────────┴────────┴──────┘
```

### PagedAttention 核心思想：KV Cache 分页管理

把 KV Cache 切成固定大小的 Block（如 16 个 token），通过 Block Table 实现逻辑到物理的映射，不连续存储，按需分配。

```
请求1 的 Block Table:
逻辑块 [0] → 物理块 7
逻辑块 [1] → 物理块 3
逻辑块 [2] → 物理块 12
```

显存利用率从 ~30% 提升到 ~96%。

### KV Cache 共享：并行解码的杀手锏

Beam Search / 并行采样时，多个分支共享 prompt 的 KV Cache，只记引用不拷贝：

```
分支1: [块A, 块B, 块C] ← 共享 prompt
       [块D]           ← 独有

分支2: [块A, 块B, 块C] ← 同一个物理块！
       [块E]           ← 独有
```

块 A/B/C 只有一个物理副本，写时复制，零额外显存。

### 效果

vLLM vs HuggingFace Transformers：
- 吞吐量：**10-24x 提升**
- 显存利用率：~30% → ~96%
- 支持更大的 batch size

## 六、连续批处理 vs 传统批处理

### 传统批处理的"木桶效应"

所有请求绑在一起同时开始同时结束，等最慢的那个：

```
请求A: 10 tokens, 早结束了但位置占着
请求D: 2000 tokens, 少数长尾请求拖死整批

算力利用率 = (10×3 + 2000) / (2000×4) = 25%
75% 的 GPU 算力在空烧
```

### 连续批处理：即进即出

每生成一个 token 就重新组批，谁结束谁退出，有新请求立刻补位：

```
Step 8:  [A→EOS] 请求A 命中结束符，立即踢出
Step 9:  [B C D E] 新请求E 立刻补入
GPU 每个 step 都在算满 batch_size 个请求
```

### 吞吐提升

```
提升倍数 ≈ max(output_len) / avg(output_len)
```

| 场景 | 提升 |
|------|------|
| 短问答（输出均匀） | 1.5-2x |
| 通用对话（长尾分布） | 3-5x |
| 混合场景（翻译+摘要+闲聊） | 8-10x |
| 极端情况（1 超长拖 N 短） | 20x+ |

业界实测（vLLM, NVIDIA Triton）：**典型 2-5x，极端 10x+**。

### 依赖技术

- PagedAttention 的块级内存管理：退出释放 Block，加入申请 Block
- 迭代级调度：每个 decode step 后立即检查并重组

## 七、量化方法对比：FP8 / INT8 / AWQ / GPTQ / SmoothQuant

### 两个维度

```
精度维度：FP8（浮点） vs INT8（整数）
策略维度：PTQ（训练后量化） vs QAT（量化感知训练）
实现维度：Weight-only vs Weight+Activation
```

### FP8 —— 浮点量化，NVIDIA 亲儿子

两种格式：
- E4M3：精度更高，适合前向
- E5M2：动态范围更大，适合反向梯度

特点：
- 浮点格式天然处理宽动态范围，对 outlier 不敏感
- H100 有硬件 FP8 Tensor Core，算力是 FP16 的 2 倍
- 几乎不需要校准
- 精度损失极小（困惑度 < 0.01）

适用：H100+ 训练和推理，必须硬件支持。

### INT8 —— 整数量化，最成熟

```
量化:   x_int8 = round(x_fp16 / scale) + zero_point
反量化: x_fp16 = (x_int8 - zero_point) × scale
```

校准方式：
- MinMax：简单但一个 outlier 毁所有
- MSE：最小化量化前后输出的均方差
- Percentile：截掉 99.99% 以上的 outlier

适用：CPU 推理、边缘部署、通用 GPU。

### SmoothQuant —— 把 outlier 平滑掉

痛点：激活值存在系统性 outlier channel（某些 channel 比其他大 10-100 倍），INT8 量化激活时小值全被截成 0。

做法：在权重和激活之间转移量化难度：

```
Y = (X × diag(s)) × (diag(s)^(-1) × W)
    └──激活变平滑──┘   └──权重吸收难度──┘

s_j = max(|X_j|)^α / max(|W_j|)^(1-α)
```

效果：实现 W8A8 全程 INT8 计算。
适用：A100 推理、需要量化激活的场景。

### GPTQ —— 逐列 Hessian 校正

Weight-only INT4 量化，核心是最优脑外科手术：

1. 对权重每列，计算 Hessian（来自校准数据）
2. 量化这一列
3. 用 Hessian 逆校正剩余未量化列，补偿误差

效果：LLaMA 7B → 显存 14GB 到 3.5GB，困惑度损失 < 0.5。
代价：量化耗时 1-4 小时。
适用：Weight-only INT4、显存第一约束场景。

### AWQ —— 保护 1% 的重要权重

核心洞察：权重中 ~1% 的显著通道贡献大部分输出，保护它们。

做法：
1. 找出 top 1% salient channels
2. 搜索每通道最优 scaling factor
3. 将 scale 吸收到权重中（weight reparameterization）

| | GPTQ | AWQ |
|------|------|-----|
| 量化依据 | Hessian（二阶） | 激活 magnitude（一阶） |
| 量化速度 | 1-4 小时 | **10-30 分钟** |
| 精度 | 高 | **略优**（0.1-0.3 ppl） |
| 推理框架 | vLLM/TGI 均支持 | vLLM/TGI 均支持 |

### 全景对比

```
                    FP8      INT8     GPTQ      AWQ     SmoothQuant
量化格式             FP8      INT8     INT4      INT4      INT8
量化对象             W+A      W+A      W only    W only    W+A
需要校准数据         否       否       需要      需要      需要
需要硬件特殊支持      H100+    通用     通用      通用      通用
量化耗时             即时      即时     1-4h      10-30min  即时
精度损失             最小      小       中-低     低        低
内存节省             2x        2x       4x        4x        2x
速度提升             2x        2x       2-3x      2-3x      2x
典型应用             训练+推理  CPU推理   单卡部署   单卡部署   A100推理
```

### 选型指南

```
有 H100，追求最高吞吐             → FP8 (W8A8)
A100 推理，需要速度               → AWQ 或 GPTQ INT4
CPU 推理                          → INT8 (W8A8)
边缘/移动端                       → INT8 或更低 bit
训练中要节省显存                   → FP8（仅 H100+）
显存放不下，精度要求高             → AWQ INT4
显存放不下，生态兼容优先           → GPTQ INT4
激活量化误差大                    → SmoothQuant + INT8
训练后立刻部署                    → AWQ（快：10min vs GPTQ 的 1-4h）
```

### 一句话总结

- **FP8**：NVIDIA 亲儿子，硬件原生支持，精度损失最小但需 H100+
- **INT8**：最通用，CPU/GPU 都支持，但激活的 outlier 是硬伤
